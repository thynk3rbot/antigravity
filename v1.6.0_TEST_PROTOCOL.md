
# LoRaLink v1.6.0 Field Test Protocol

**Target Release:** v1.6.0 (Range Boost + Power-Miser)
**Date:** 2026-03-08

## 🧪 Overview

This protocol is designed to verify the performance gains of the packed binary protocol and the reliability of the integrated Smart Ag power management.

---

## 🚦 Test Environment Setup

1. **Master Node:** Connected via USB to PC (Serial Monitor open at 115200). IP: `172.16.0.27`.
2. **Slave Node:** Battery powered (Solar setup). IP: `172.16.0.26`.
3. **Distance:** Start at 100m, move to 1km+, looking for the "Digital Cliff" where legacy packets fail but boosted packets survive.

---

## 🛠️ Step 1: Range Boost & ToA Verification

*Goal: Confirm that variable-sized packets reduce Time-on-Air (ToA) and improve reliability at the edge.*

1. **In the Master console, type:** `RTEST`
2. **Observe Serial Logs:**
   - Look for `LORA: Legacy-TX (92 bytes, ~327ms ToA)`
   - Look for `LORA: Var-TX (16-30 bytes, <180ms ToA)`
3. **Analyze:** Record the SNR and RSSI of the responding node.
   - If Legacy SNR is below -15dB but Boosted packets are still being acknowledged, the protocol is a success.

---

## 🛠️ Step 2: Reliable Binary Transport (ACK)

*Goal: Confirm that the manual command layer handles drops gracefully.*

1. **In the Master console, type:** `BINCMD 6 1` (Binary Ping)
2. **Observe Serial Logs:**
   - Source: `LORA: Queued reliable BINARY -> ALL (0x06)`
   - Destination: `BIN: Compact RX (4 bytes)` -> `BIN: ACK token 0x06`
   - Source: `LORA: BINARY ACK Verified from ALL`
3. **Chaos Test:** Briefly cover the Master's antenna or step behind a wall during the 3-second window.
   - Verification: Source should log `LORA: Resending reliable command (Try 1/3)...`

---

## 🛠️ Step 2.1: Protocol Failover Algorithm

*Goal: Verify that the system automatically falls back to Legacy Text when Binary fails.*

1. **In the Master console, type:** `FPING SLAVE`
2. **Simulate Total Binary Failure:** Use a physical shield (metal box) or extreme distance to ensure the Binary ACK is never received.
3. **Observe Serial Logs:**
   - Source: `LORA: Queued FAILOVER PING -> SLAVE`
   - Source: Should attempt 3 retries: `LORA: Resending reliable command (Try 3/3)...`
   - **Failover Trigger**: Source should log `LORA: Protocol Failover -> Trying Legacy PING for SLAVE`.
   - Source: Should then send a text-based `PING` and log `LORA: Sent (Boosted) Text PING to SLAVE`.
4. **Final Verification**: If the Slave is still reachable via Text, the Master logs `LORA: TX Text Success`. If it fails, `Delivery failed -> SLAVE`.

---

## 🛠️ Step 3: Power-Miser (Tier Logic)

*Goal: Confirm the system correctly gates peripherals and slows heartbeats when low on power.*

1. **Force Critical Mode:** `PMISER CRITICAL`
2. **Verify Slave State:**
   - [ ] OLED screen should turn OFF immediately.
   - [ ] Dashboard (if connected via WiFi) should show "Mode: CRITICAL".
   - [ ] Type `SCHED` and verify long heartbeat interval (3600s).
3. **Force Normal Mode:** `PMISER NORMAL`
4. **Verify Slave State:**
   - [ ] OLED screen should turn ON.
   - [ ] Dashboard should show "Mode: NORMAL".
   - [ ] Type `SCHED` and verify standard heartbeat interval (300s).

---

## 🛠️ Step 4: Peripheral Gating (Hardware)

*Goal: Verify VEXT control shuts down auxiliary sensors.*

1. On the Slave Node, check the `VEXT` rail (if using a voltmeter or observing a connected sensor LED).
2. **Verify:** In `CONSERVE` or `CRITICAL` mode, the `VEXT` rail should be HIGH (Heltec V3 VEXT is Low-Enable, so HIGH = Rail OFF).
