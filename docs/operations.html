⚙️ System Operations

Managing the Magic PC platform — service orchestration, supervised startup, and crash recovery.

Service Architecture

The Magic platform runs five coordinated services on a single Windows host. No containers or cloud dependencies are required.

ServicePortDescription
MQTT Broker1883Mosquitto — local message bus for device telemetry
Magic Daemon8001FastAPI — REST API, fleet management, MQTT client
Fleet Dashboard8000Web UI — device monitoring, OTA firmware flash
Magic Messenger8400PWA mesh chat — WebSocket bridge to LoRa network
AI Assistant8300Local AI with mesh context and device awareness

One-Click Launch

To start the entire Magic stack with a single click, use the root-level batch file:

Start_Magic.bat

This performs the following sequence:

  1. Starts Mosquitto MQTT broker (port 1883)
  2. Launches Magic Daemon with service manager (ports 8001/8000)
  3. Starts Magic Messenger bridge (port 8400)
  4. Opens Fleet Dashboard in the default browser

Individual service launchers are available in tools/ if you need to start them separately.

Background Mode (Supervised)

For headless deployments where services must restart automatically after a crash:

python tools/start_bg_services.py

This runs a supervisor process that monitors each service in a background thread. If any service exits unexpectedly, it is restarted after a 5-second cooldown. All output is written to the logs/ directory:

Log FileService
logs/daemon.logMagic Daemon
logs/webapp.logFleet Dashboard
logs/loramsg.logMagic Messenger
logs/assistant.logAI Assistant

Stop the supervisor (and all managed services):

python tools/start_bg_services.py stop

Automated Startup (Reboot Survival)

To register Magic as a Windows login-time startup task:

⚠️ Admin Rights Required

Registering a startup task requires Administrator privileges.

  1. Navigate to the tools/ directory.
  2. Right-click Register_Startup_Task.bat and select "Run as Administrator".
  3. Magic is now registered in Windows Task Scheduler to launch on login.

Protocol Failover (LMX Transport Selection)

The LMX mesh protocol selects the best available transport for each outbound message using a three-tier fallback ladder:

  1. WiFi HTTP — used if the destination node has an active IP (from its NODE_ANNOUNCE capability advertisement).
  2. ESP-NOW — used if the destination is a known neighbor with ESP-NOW capability in the neighbor table (3-minute TTL).
  3. LoRa — always-available fallback for any reachable node in the mesh.

Nodes broadcast their capabilities (WiFi, ESP-NOW, BLE, LoRa) via NODE_ANNOUNCE packets every 60 seconds. The daemon aggregates these into the fleet device registry.

MQTT Topic Contract

All device-to-daemon communication follows a fixed topic structure:

TopicDirectionPayload
magic/{node_id}/telemetryDevice → DaemonJSON: battery, RSSI, uptime, temp
magic/{node_id}/statusDevice → DaemonJSON: connectivity, capabilities
magic/{node_id}/cmdDaemon → DevicePlain text command (e.g. RESTART, STATUS)

Requirements

  • Python 3.10+ in PATH — pip install -r daemon/requirements.txt
  • Mosquitto installed — winget install mosquitto
  • PlatformIO for firmware builds — pio run -t upload -e heltec_v4
