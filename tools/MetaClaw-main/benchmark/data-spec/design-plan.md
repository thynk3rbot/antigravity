# MetaClaw Evolution Benchmark — Design Plan

## 1. 目标

验证 MetaClaw 的自演化策略（skill evolution + RL 训练）能否使 agent 在重复类似任务上逐步提升，而基线推理引擎（无演化）无法做到。

---

## 2. 核心设计原则：隐式偏好任务

每个任务满足以下条件：

- **首次必然失败**：workspace 和任务描述刻意缺失某个关键规则/偏好，LLM 凭通用经验会给出"合理但错误"的答案
- **可规则判定**：正确性可由代码自动验证，无需人工打分
- **反馈即训练信号**：每轮任务执行后，用户发出的反馈（对/错及说明）被 MetaClaw 拦截，成为 skill 演化和 RL 的训练材料
- **跨天泛化**：同类偏好以不同措辞/场景出现在后续 scenario（rephrase），base model 总是重复犯错，MetaClaw 学会后可以迁移

这一设计反映真实用户行为：用户往往第一次说不清需求，在纠错过程中逐渐塑造 agent 的行为模式。

---

## 3. 任务领域

覆盖多个日常工作场景，不局限于代码任务：

| 领域 | 示例任务 |
|------|---------|
| 文档与写作 | 整理会议纪要、起草邮件、更新周报 |
| 数据处理 | 解析日志、生成报表、转换文件格式 |
| 代码工程 | 写单元测试、修 bug、重构函数 |
| 项目管理 | 拆解任务、更新看板、估算优先级 |
| 日程与沟通 | 安排会议、回复消息、整理 action items |

隐式偏好同样横跨多领域，如："邮件回复必须带 `[ACTION REQUIRED]` 前缀"、"所有时间用 ISO 8601"、"金额超过 5000 需要附说明"。

---

## 4. 跨天学习弧（Learning Arc）

30 个 scenario 按"弧"组织，**6 弧 × 5 天**，每弧引入 1 个新偏好规则，已引入的规则在后续弧持续出现。同一弧的 scenario 放在一起造数据，保证内部一致性。

| 弧 | 天 | 引入偏好 | 延续偏好 |
|----|-----|---------|---------|
| A  | 1–5  | P1 | — |
| B  | 6–10 | P2 | P1 |
| C  | 11–15 | P3 | P1, P2 |
| D  | 16–20 | P4 | P1–P3 |
| E  | 21–25 | P5 | P1–P4 |
| F  | 26–30 | — | P1–P5 混合收尾 |

每弧第 1 天首次触碰新偏好（首次必然失败），第 3/5 天换场景复测同类规则，验证迁移能力。

---

## 5. Scenario 与 Round 结构

- **30 个 scenario**，每个对应一天的交互（一个 session 文件）
- **每 scenario 10–20 个 eval round**，多数 10 轮，复杂场景至多 20 轮
- Session 初始上下文：极简，仅 1–2 轮引导对话（介绍用户角色、当天工作背景、workspace 结构），不含任务内容

Round 注入机制：
```
Round 1:   [任务]
Round k:   [上轮 feedback（对/错 + 说明）] + [新任务]
最终消息:  [最后一轮的 feedback，无新任务]（standalone user 消息）
```

每个 round 定义两个 feedback 模板（正确 / 错误），runner 根据评分结果选择注入。

---

## 6. 评测协议

### Baseline 模式
- 30 个 scenario **并行独立**运行，普通 LLM，无 MetaClaw
- workspace 每次从源目录复制，保证隔离

### MetaClaw 模式
- 30 个 scenario **严格串行**，Day N 结束 → RL 训练 → Day N+1
- 每天结束记录 skill 快照供复现

### 核心指标

| 指标 | 含义 |
|------|------|
| `overall_pass_rate` | 全部 rounds 通过率 |
| `first_encounter_rate` | 每类偏好"首次遇见"通过率 |
| `repeat_encounter_rate` | 同类偏好"后续遇见"通过率 |
| `delta = repeat − first` | 学习效果量化 |
| `arc_pass_rate` | 按弧分组通过率，展示学习曲线 |

---

## 7. 偏好类别（待细化，造数据时按弧展开）

| 代号 | 类别 | 示例规则 |
|------|------|---------|
| P1 | 输出格式 | 时间用 ISO 8601；JSON 禁止 null 值；列表用 `-` 不用数字编号 |
| P2 | 命名与结构 | 文件名 snake_case + 日期前缀；函数注释用中文；字段顺序固定 |
| P3 | 字段完整性 | 报告必含 `confidence`；任务卡必含 `priority`；邮件必含主题标签 |
| P4 | 工作流约束 | 写入前先读取做 diff；操作前附 dry-run 结果；超阈值需 flag |
| P5 | 沟通风格 | 邮件回复三段式；反馈带具体行号；总结不超过 3 条 |

实际造数据时可合并或替换，保证每弧的偏好在对应任务领域内自然。
